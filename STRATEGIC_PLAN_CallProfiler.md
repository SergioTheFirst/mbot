# CALL PROFILER — Генеральный план

## Вердикт совета экспертов

Проект технически реализуем на заявленном железе, но предыдущее проектирование содержит ряд архитектурных переусложнений, которые убьют проект на старте. Ниже — выверенный план, очищенный от шума.

---

## Стратегические принципы

**Принцип 1: Вертикальный срез, а не горизонтальные слои.**
Не строить «весь парсер», потом «весь Whisper», потом «всю LLM». Вместо этого — провести один звонок от записи до Telegram-сообщения. Это даёт работающий продукт на каждом этапе.

**Принцип 2: Отложить всё, без чего можно жить.**
Knowledge Graph (Neo4j), BS Index, Context Cards, Overlay — всё это фазы 3–5. Фаза 1 — это «файл вошёл → текст вышел → саммари в Telegram».

**Принцип 3: Одна база, один язык, один процесс.**
SQLite на MVP, PostgreSQL на проде. Python. Никаких LangChain/LlamaIndex — они добавят непредсказуемость и абстракции, которые невозможно отлаживать. Прямые вызовы Ollama API.

**Принцип 4: GPU — узкое горлышко. Уважать очередь.**
RTX 3060 12GB не потянет Whisper и LLM одновременно. Строго последовательный пайплайн: сначала Whisper, потом LLM. Очередь на файловой системе (watchdog + SQLite-статусы), без Redis/Celery на MVP.

---

## Критические поправки к исходному проекту

**1. Переусложнение стека.** В обсуждениях фигурируют Neo4j + PostgreSQL + ChromaDB + Redis + Celery + LangChain. На RTX 3060 с 16GB RAM это не взлетит — каждый компонент жрёт память, а GPU уже занят моделями. На MVP: SQLite + Ollama + faster-whisper. Всё.

**2. «4-Agent Pipeline» — роскошь.** Четыре последовательных прогона LLM 14B на каждый звонок = 4× время обработки. На MVP достаточно одного хорошо структурированного промпта с JSON-выходом. Агентную декомпозицию вводить только когда один промпт начнёт проседать по качеству.

**3. Knowledge Graph — не MVP.** Граф требует консистентного связывания сущностей между звонками (entity resolution). Это отдельная исследовательская задача. На MVP: простые таблицы `calls`, `contacts`, `action_items` с foreign keys.

**4. RAM 16GB — на грани.** Qwen 2.5 14B-Q4 ≈ 10GB RAM, faster-whisper large-v3 ≈ 3GB VRAM. Они не помещаются вместе. Решение: загружать модели последовательно (Whisper → выгрузить → загрузить LLM → выгрузить). Или: Whisper на GPU, LLM через Ollama с автоматической выгрузкой.

**5. Overlay / CardDAV — преждевременно.** Пока нет работающего ядра анализа, оптимизировать доставку на телефон бессмысленно. Telegram-бот покрывает 90% потребностей.

## Фазы разработки

### ФАЗА 0 — Инфраструктура (3–5 дней)

**Цель:** Развёрнутая среда, в которую можно кидать аудиофайлы и получать результат.

**Шаги:**

1. Установить Ollama на Windows, загрузить модель `qwen2.5:14b-instruct-q4_K_M`. Проверить, что инференс работает через `ollama run`. Замерить скорость (tokens/sec) и потребление RAM.

2. Установить `faster-whisper` (через pip). Скачать модель `large-v3`. Прогнать тестовый русскоязычный аудиофайл. Замерить время транскрибации на минуту аудио.

3. Создать структуру проекта:
   ```
   call_profiler/
   ├── config.py          # Все пути, параметры моделей
   ├── db/
   │   ├── schema.sql     # Схема SQLite
   │   └── repository.py  # CRUD-операции
   ├── pipeline/
   │   ├── watcher.py     # Мониторинг входящей папки
   │   ├── ingest.py      # Парсинг имени файла + запись в БД
   │   ├── transcribe.py  # Whisper
   │   ├── diarize.py     # Pyannote (или channel split)
   │   ├── analyze.py     # Вызов LLM
   │   └── orchestrator.py# Последовательный запуск шагов
   ├── bot/
   │   └── telegram.py    # Telegram-бот
   ├── tests/
   └── main.py
   ```

4. Создать SQLite-базу с минимальной схемой:
   - `contacts` (id, phone_e164, display_name, created_at, updated_at)
   - `calls` (id, contact_id, direction, datetime, duration_sec, file_path, status)
   - `transcripts` (id, call_id, speaker, start_sec, end_sec, text)
   - `analyses` (id, call_id, priority, risk_score, summary, action_items_json, flags_json, raw_llm_json, created_at)

5. Настроить на телефоне автозапись + FolderSync → SFTP на сервер в папку `incoming/`.

**Критерий завершения:** SFTP принимает файлы, Whisper транскрибирует тестовый файл, Ollama отвечает на промпт, БД создана.


### ФАЗА 1 — Сквозной пайплайн (7–10 дней)

**Цель:** От аудиофайла до сообщения в Telegram — автоматически, без участия человека.

**Шаги:**

1. **Парсер имён файлов** (`ingest.py`). Поддержка форматов BCR (`date_direction_number_name.ext`) и формата в скобках. Нормализация номера в E.164 (+7...). Создание/обновление контакта в БД. Юнит-тесты на 20+ грязных имён файлов.

2. **Конвертация аудио** (ffmpeg → WAV 16kHz mono). Обёртка, которая принимает любой формат и отдаёт стандартизированный WAV.

3. **Транскрибация** (`transcribe.py`). faster-whisper large-v3. Выход: список сегментов `{start, end, text, language}`. Сохранение в таблицу `transcripts`.

4. **Диаризация** (`diarize.py`). Два подхода (выбрать по результатам тестов):
   - **Вариант А:** Если рекордер пишет стерео (левый канал = я, правый = собеседник) — простое разделение каналов через ffmpeg. Дёшево и надёжно.
   - **Вариант Б:** Pyannote-audio для моно-записей. Учесть: модель требует HuggingFace-токен и ~1.5GB VRAM. Не запускать одновременно с Whisper.

5. **Анализ LLM** (`analyze.py`). Один промпт → один JSON. Промпт получает: полную стенограмму с метками спикеров, метаданные (номер, имя контакта, дата, длительность). Возвращает: `{priority: 0-100, risk_score: 0-100, summary: str, action_items: [...], flags: {...}}`. Результат сохраняется в `analyses`.

6. **Оркестратор** (`orchestrator.py`). Watchdog следит за `incoming/`. При появлении файла: проверяет целостность (ждёт, пока размер стабилизируется) → запускает шаги последовательно → обновляет статус в `calls.status` (new → transcribing → analyzing → done / error).

7. **Telegram-бот** (`telegram.py`). При каждом обработанном звонке отправляет саммари. Команда `/digest N` — топ-N по priority за последние дни. Команда `/search текст` — полнотекстовый поиск по стенограммам (SQLite FTS5).

**Критерий завершения:** Положил аудиофайл на SFTP → через 2–5 минут получил саммари в Telegram. Работает для 10 последовательных файлов без сбоев.


### ФАЗА 2 — Стабилизация и повседневная эксплуатация (5–7 дней)

**Цель:** Система работает 24/7 без присмотра.

**Шаги:**

1. **Обработка ошибок.** Каждый шаг пайплайна обёрнут в try/except с логированием. При сбое: файл помечается status=error, система переходит к следующему. Периодический ретрай ошибок.

2. **Дедупликация.** MD5-хеш аудиофайла перед обработкой. Если хеш уже в БД — пропуск. Это Tier 1. Tier 2 (по audio fingerprint) — отложить.

3. **Утренний дайджест.** Cron/scheduled task в 09:00: собрать незакрытые action_items, высокие risk_score за вчера, непрочитанные звонки. Отправить в Telegram.

4. **Обратная связь.** Inline-кнопки в Telegram: [OK] / [Неточно] / [Отложить]. Сохранять реакции в БД для будущей калибровки промптов.

5. **Бэкап.** Ежедневное копирование SQLite-базы и папки аудио на второй диск (robocopy / rsync через WSL). Ротация: хранить 7 ежедневных + 4 еженедельных копии.

6. **Мониторинг очереди.** Если файлов в очереди > 10 — уведомление в Telegram «Система перегружена, задержка ~X мин».

**Критерий завершения:** Система работает неделю в реальных условиях без ручного вмешательства.


### ФАЗА 3 — Улучшение качества анализа (2–3 недели)

**Цель:** Система даёт аналитику, ради которой всё затевалось.

**Шаги:**

1. **Контекст предыдущих звонков.** При анализе текущего звонка — LLM получает саммари последних 3–5 звонков с этим контактом. Это позволяет ловить противоречия и отслеживать обещания.

2. **Risk Index v1.** Расширить промпт: LLM должен обосновывать risk_score цитатами из стенограммы. Три оси: Vagueness (размытость), Defensiveness (защитная риторика), Consistency (совпадение с историей).

3. **Полнотекстовый поиск.** Включить SQLite FTS5 на таблице транскриптов. Команда `/search ключевое слово` в Telegram.

4. **Трекинг обещаний.** Отдельная таблица `promises` (contact_id, call_id, promise_text, due_date, status). LLM извлекает обещания из стенограммы. Утренний дайджест включает просроченные обещания.

5. **Калибровка промптов.** На основании накопленной обратной связи (кнопки [Неточно]) — доработка system-промпта. Это ручная работа, не автоматизация.

**Критерий завершения:** Risk Index выдаёт осмысленные оценки с обоснованиями. Поиск по стенограммам работает. Обещания трекаются.


### ФАЗА 4 — Расширение интерфейсов (2–3 недели)

**Цель:** Информация доступна в нужный момент, а не только по запросу.

**Шаги:**

1. **Веб-интерфейс (минимальный).** FastAPI + простой HTML. Таблица звонков с фильтрами (контакт, дата, risk_score). Карточка звонка: саммари, стенограмма, аудиоплеер. Это не приоритет, но для ретроспективного анализа — незаменимо.

2. **Overlay на телефоне (Вариант C из консенсуса).** Сервер генерирует текстовые файлы `CallNotes/{phone_e164}.txt` → FolderSync доставляет на телефон → MacroDroid показывает overlay при входящем. Работает со второго звонка.

3. **Автоконтакты (опционально).** Для номеров, с которых звонили > 2 раз: создавать контакт через CardDAV/Radicale. Для остальных — не засорять книгу.

4. **API для интеграций.** REST-эндпоинты: GET /calls, GET /calls/{id}, GET /contacts/{id}/history. Это фундамент для любых будущих клиентов.

**Критерий завершения:** Веб-интерфейс показывает историю. Overlay работает на телефоне.


### ФАЗА 5 — Продвинутая аналитика (долгосрочно)

**Цель:** Система, которая «понимает» контекст на уровне CRM.

**Шаги (выполнять только после стабильной работы фаз 1–4):**

1. **Векторный поиск.** ChromaDB или pgvector. Эмбеддинги стенограмм для семантического поиска («Кто обсуждал поставку труб?»).

2. **Knowledge Graph.** PostgreSQL + Apache AGE или простые таблицы связей. Сущности: Person, Company, Deal. Связи: WORKS_AT, PROMISED, BLAMED.

3. **Multi-agent pipeline.** Разделение анализа на Extractor (сущности) → Analyst (сравнение с историей) → Secretary (формат выхода). Только если один промпт перестаёт справляться.

4. **Паралингвистика.** Анализ тона голоса (pitch, energy) как дополнительный сигнал для risk_score. Экспериментальная функция.

5. **Профили контактов.** Накопительный «портрет» каждого контакта: стиль общения, паттерны поведения, история обещаний. Обновляется после каждого звонка.

## Архитектура обработки одного звонка

```
Телефон (запись) 
    ↓ FolderSync/SFTP
Сервер: incoming/
    ↓ watchdog (orchestrator.py)
[1] Проверка: файл полный? дубликат?
    ↓
[2] Парсинг имени файла → метаданные (номер, дата, направление)
    ↓
[3] ffmpeg → WAV 16kHz mono
    ↓
[4] faster-whisper → сегменты текста с таймкодами
    ↓
[5] Диаризация (стерео-сплит или pyannote)
    ↓
[6] Сборка стенограммы: [00:15] Я: ... [00:18] Собеседник: ...
    ↓
[7] Подгрузка контекста: саммари последних N звонков с этим номером
    ↓
[8] Ollama (Qwen 14B) → JSON: {priority, risk_score, summary, action_items, flags}
    ↓
[9] Сохранение в SQLite (calls, transcripts, analyses)
    ↓
[10] Telegram: отправка саммари
    ↓
[11] Генерация CallNotes/{номер}.txt (для overlay, фаза 4)
```

**Время обработки (оценка на RTX 3060):**
- Шаги 1–3: ~5 сек
- Шаг 4 (Whisper): ~30–60 сек на минуту аудио
- Шаг 5 (диаризация): ~10–20 сек (стерео) / ~30–60 сек (pyannote)
- Шаги 7–8 (LLM): ~20–40 сек
- **Итого: ~1–3 мин на средний 5-минутный звонок**

## Технологический стек (окончательный для MVP)

| Компонент | Решение | Обоснование |
|-----------|---------|-------------|
| OS | Windows 10/11 | Текущая среда, Docker не нужен на MVP |
| Python | 3.11+ | Основной язык |
| Транскрибация | faster-whisper (large-v3) | Быстрее оригинального Whisper в 4×, меньше VRAM |
| Диаризация | ffmpeg channel split (стерео) или pyannote (моно) | Стерео = бесплатно и точно |
| LLM | Ollama + Qwen 2.5 14B Q4_K_M | Лучшее соотношение качество/размер для русского |
| БД | SQLite + FTS5 | Нулевая настройка, бэкап = cp файла |
| Очередь | watchdog + статусы в SQLite | Без внешних зависимостей |
| Бот | python-telegram-bot | Стабильная библиотека |
| Синхронизация | FolderSync + SFTP (OpenSSH на Windows) | Уже работает |
| Бэкап | robocopy / scheduled task | Встроено в Windows |

**Что НЕ входит в MVP и почему:**
- Docker — добавит сложность без выгоды для одного пользователя
- Redis/Celery — SQLite-очередь достаточна для десятков звонков в день
- PostgreSQL — оверкилл для одного пользователя, переход позже тривиален
- Neo4j — отдельная инфраструктура, не нужна до фазы 5
- ChromaDB — семантический поиск не нужен до фазы 5
- LangChain — абстракция, скрывающая суть; прямые HTTP-вызовы к Ollama проще и предсказуемее

## Риски и контрмеры

| Риск | Вероятность | Контрмера |
|------|-------------|-----------|
| Whisper плохо распознаёт русскую речь по телефону | Средняя | Тестировать на реальных записях до начала разработки. Альтернатива: whisper-large-v3-turbo |
| 16GB RAM не хватает для Whisper + LLM | Высокая | Последовательная загрузка моделей. Ollama `--num-gpu 0` для CPU-fallback LLM, если Whisper на GPU |
| Формат записей отличается от ожидаемого | Средняя | Сначала собрать 20 реальных файлов, изучить форматы имён и аудио, только потом писать парсер |
| Pyannote требует GPU одновременно с Whisper | Высокая | Стерео-запись устраняет проблему. Если моно — последовательная обработка |
| Windows перезагружается ночью (обновления) | Высокая | Отключить автообновления. Настроить автозапуск через Task Scheduler |
| LLM галлюцинирует action_items | Средняя | Промпт с примерами (few-shot). Обратная связь через Telegram-кнопки |
| Переполнение диска аудиозаписями | Низкая | Мониторинг свободного места + архивация на HDD старше 30 дней |

## Порядок первых действий (первые 48 часов)

1. Собрать 10–20 реальных аудиозаписей звонков с телефона
2. Изучить форматы: имена файлов, кодеки, моно/стерео, длительности
3. Установить faster-whisper, прогнать все 20 файлов, оценить качество
4. Установить Ollama + Qwen 14B, написать тестовый промпт с одной стенограммой
5. Замерить: RAM, VRAM, время на файл, качество выхода
6. На основе замеров — принять решение: размер модели Whisper, размер модели LLM, последовательная или параллельная загрузка

**Только после этих замеров начинать писать код пайплайна.**
